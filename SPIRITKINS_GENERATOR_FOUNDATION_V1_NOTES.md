# Spiritkins Generator Foundation V1

## Architecture

The generator foundation is an orchestration layer, not a model-execution layer.

Current components:

- `src/services/spiritkinGeneratorService.mjs`
  - authoritative generator store and workflow service
  - persists generator records to `runtime_data/spiritkin_generator_foundation_v1.json`
- `src/routes/admin.mjs`
  - internal admin endpoints for generator summary, job creation, and review actions
- `src/routes/spiritkins.mjs`
  - enriches canonical Spiritkin payloads with runtime media assignments from the generator store
- `spiritkins-app/command-center.js`
  - internal command-center generator UI
- `spiritkins-app/command-center.css`
  - generator form, queue, and attachment styling

## Data Model

The foundation store tracks:

- `imageJobs`
  - structured image generation requests
- `videoJobs`
  - structured trailer / video generation requests
- `outputs`
  - output slot records with provider state, review state, canonical flag, runtime-attach flag, and storage paths
- `assignments`
  - per-Spiritkin canonical/runtime media slot attachment state
- `reviewLog`
  - append-only review actions and notes

Each job contains:

- Spiritkin identity target
- owner type
  - canonical
  - user_created
- slot name
- target audience
- entitlement gate
- structured spec
- prompt package
- negative prompt package
- provider status
- version tag

Each output contains:

- output id
- job id
- media kind
- slot name
- storage paths
  - draft
  - approved
  - rejected
- review status
- canonical flag
- runtime attached flag
- review history

## Generation Flow

### Image Flow

Input fields:

- Spiritkin name
- archetype / class
- colors
- element / theme
- mood / personality
- pose
- environment
- render style
- rarity / tier

Generated by the foundation:

- structured generation spec
- prompt package
- negative prompt package
- output slot record
- version tag
- storage path plan
- provider status `awaiting_provider`

### Video Flow

Input fields:

- Spiritkin name
- trailer type
- duration
- shot style
- script / voice line
- music mood
- attached assets

Generated by the foundation:

- structured trailer spec
- shot list
- prompt package
- negative prompt package
- output slot record
- media slot target
- version tag
- storage path plan
- provider status `awaiting_provider`

## Review Flow

Current command-center review actions:

- approve
- reject
- mark canonical
- attach to runtime

Review actions write:

- output review history
- generator review log
- canonical media assignment state when applicable
- per-slot assignment history for the Spiritkin

## Asset Storage and Versioning

The foundation models storage in three categories for every output:

- draft
- approved
- rejected

The current system stores:

- the record
- the storage intent/path plan
- the prompt/spec history

It does not yet execute uploads or render files.

## Premium User Expansion Readiness

The current foundation already supports:

- canonical Spiritkins
- future user-created Spiritkins
- role-based gating fields
- per-slot media attachments that do not assume only the core cast
- runtime attachment model reusable for premium creation later

The current request model includes:

- `ownerType`
- `ownerId`
- `targetAudience`
- `entitlementGate`

That is the main compatibility layer for future premium rollout.

## What Still Requires Provider / Model Integration

Not implemented yet:

- actual image provider execution
- actual video provider execution
- artifact upload / blob storage
- artifact path completion from real generated files
- admin preview thumbnails/video playback for generated outputs
- user-facing premium generator surface
